# 《回声病房》Game Vision / 游戏愿景文档

> **文档版本：** 0.1（概念立项版）  
> **项目代号：** Project Echo Ward  
> **游戏类型：** 第一人称 3D 心理恐怖、探索解谜、环境叙事  
> **目标平台：** Windows PC 单机  
> **核心引擎候选：** Panda3D 1.10.16 + Python 3.11  
> **文档日期：** 2026-07-14  
> **文档状态：** 前期设计，可在后续概念评审中调整

---

## 1. 项目概述

### 1.1 一句话概念

玩家在一所早已废弃的医院中醒来。每当他穿过住院部走廊尽头的防火门，医院就会向事故发生前倒退一天；与此同时，一个在黑暗中巡游的“值夜护士”正在逐渐记住玩家每一轮循环中的行为。

### 1.2 核心体验

《回声病房》希望提供一种以**不确定性、脆弱感和记忆错位**为核心的第一人称心理恐怖体验。游戏不依赖连续跳脸、密集追逐或大量血腥画面，而是通过空间变化、声音误导、环境叙事和逐步显现的规则，让玩家持续怀疑：

- 眼前的医院是否真实；
- 自己是否曾经来过这里；
- 怪物是在追逐玩家，还是在重复玩家过去的行为；
- 主角究竟是事故受害者、旁观者，还是责任人。

### 1.3 目标玩家

本作面向喜欢以下体验的玩家：

- 第一人称心理恐怖；
- 封闭空间探索；
- 环境叙事与碎片化剧情；
- 观察型谜题；
- 有限追逐但不以战斗为主；
- 能够在一到三次游玩中理解完整真相的短篇独立游戏。

### 1.4 初步规模

| 项目 | 目标范围 |
|---|---|
| 首次通关时长 | 60～90 分钟 |
| 完整收集时长 | 90～150 分钟 |
| 主要地图 | 一座三层医院 |
| 核心谜题 | 5～7 个 |
| 主要追逐段落 | 2～3 段 |
| 主要怪物 | 1 个 |
| 环境异象类型 | 10～20 组，可复用组合 |
| 结局 | 3 个 |
| 战斗系统 | 无直接战斗，以躲避和暂时阻断为主 |
| 存档方式 | 自动存档 + 章节检查点 |

## 2. 创作愿景

### 2.1 我们想做什么

本项目希望做一款规模可控但氛围完整的 3D 单机恐怖游戏。玩家不是在一个巨大的开放世界里收集任务，而是在有限、逐渐变化的医院空间中反复探索。相同地点在不同循环中呈现出细微或剧烈变化，使玩家从“记住地图”逐步转向“怀疑地图”。

游戏希望让玩家在通关后仍然回想：某些声音究竟是什么、某个房间是否真的出现过、怪物是不是另一个循环中的自己，以及主角为何一直不愿打开 404 号病房。

### 2.2 我们不想做什么

本项目不以以下方向为核心：

- 不做依靠高频跳脸维持刺激的惊吓合集；
- 不做以枪械射击和击杀怪物为中心的动作游戏；
- 不做大规模开放世界；
- 不做大量重复搜集物资的生存制造系统；
- 不做过度复杂的背包管理；
- 不用超长文字直接解释全部剧情；
- 不把随机生成当作关卡设计的替代品；
- 不追求超出个人或小团队能力范围的 3A 写实规格。

### 2.3 预期玩家情绪曲线

```text
陌生与疑惑
→ 短暂安全
→ 发现细微异常
→ 怀疑自身记忆
→ 学习环境规则
→ 第一次遭遇真实威胁
→ 获得局部掌控感
→ 规则被打破
→ 高压追逐与真相揭示
→ 做出道德选择
→ 余韵与重新解读
```

游戏应避免长时间保持同一强度。恐怖来源于“安静之后会发生什么”，因此需要有意识地安排低压探索、信息发现、假警报、真实威胁和释放阶段。

## 3. 核心设计支柱

### 3.1 支柱一：熟悉空间中的不确定性

医院的总体结构应当足够清晰，让玩家能够形成空间记忆；但每次循环都会改变部分细节：

- 门牌号码变化；
- 房间内部位置发生错位；
- 同一条走廊长度不同；
- 出现不属于当前年代的物品；
- 墙上照片中的人物数量变化；
- 已经打开的门重新上锁；
- 某些安全路线变为危险路线；
- 玩家上一轮移动过的物品被“另一个人”重复移动。

变化必须具有设计规律，不应完全随机。玩家可以逐渐理解规则，却无法获得绝对安全感。

### 3.2 支柱二：声音先于视觉

声音是游戏最主要的威胁信息来源。玩家往往先听到异常，再决定是否前进、躲藏或关闭灯光。

关键声音包括：

- 输液瓶有规律的玻璃碰撞声；
- 墙后、楼上和通风管中的脚步；
- 远处广播的错误楼层提示；
- 门外停下的呼吸；
- 不符合玩家动作节奏的第二组脚步；
- 怪物接近时环境底噪突然消失；
- 电话或录音机重播玩家之前听过的内容。

音频必须支持方向性和距离衰减，使玩家能依靠听觉做出判断，而不是只依赖背景音乐。

### 3.3 支柱三：玩家脆弱但并非无能

玩家无法杀死主要怪物，但能够通过观察和操作提高生存概率：

- 缓慢行走降低噪声；
- 关闭手电筒避免暴露；
- 关门延缓追击；
- 躲入柜子或床下；
- 利用广播、闹钟或医疗设备制造声源；
- 切断局部电源改变怪物路线；
- 使用一次性闪光设备短暂击退怪物；
- 记住安全房间和逃生通道。

玩家失败应主要源于误判规则，而不是怪物作弊或毫无预警地出现。

### 3.4 支柱四：观察会改变世界

视线本身是重要的交互机制：

- 某些物体只在玩家移开视线后移动；
- 镜中影像与现实存在时间差；
- 特定文字只在手电筒关闭后显现；
- 长时间注视异常物会触发不同事件；
- 摄像机画面显示现实中不可见的通道；
- 玩家必须决定是否回头确认声音来源。

这一支柱要求游戏准确记录玩家视线、停留时间和观察对象，并将其用于事件触发。

### 3.5 支柱五：真相由玩家拼接

剧情信息不会由单一角色一次性说明。玩家通过以下内容自行建立事件时间线：

- 病历和用药记录；
- 护士值班表；
- 事故调查文件；
- 监控录像；
- 录音带；
- 患者绘画；
- 医疗设备上的时间；
- 房间布置的年代差异；
- 主角对特定物品的非自愿反应。

玩家即使不收集全部线索，也能完成表层逃生目标；但只有理解关键证据，才能进入完整结局。

## 4. 世界观与剧情方向

### 4.1 表层设定

一所名为“澄安医院”的私人医院在多年前发生火灾和医疗事故，此后被永久封闭。主角在医院 404 号病房醒来，没有明确记忆，只知道必须找到出口。

医院中的钟表、病历和广播不断显示不同日期。主角穿过住院部防火门后，会进入事故前一天、前两天或更早的医院状态，但部分物品仍保留上一轮的变化。

### 4.2 深层设定

主角曾是医院的一名医生，并参与过一项未经充分告知的实验性治疗。事故发生当晚，主角在服从命令、保护患者和掩盖责任之间做出了错误选择。医院循环并非普通闹鬼，而是由患者记忆、主角创伤和某项医疗装置共同形成的精神空间。

“值夜护士”可能不是传统意义上的实体怪物。它既代表事故中死亡的护理人员，也包含主角对责任的逃避。随着循环推进，它会越来越像主角本人。

### 4.3 主题

本作关注以下主题：

- 记忆是否等于事实；
- 逃避责任能否真正保护自己；
- 制度命令与个人道德之间的冲突；
- 受害者叙事如何被责任人重构；
- 承认错误是否仍有意义。

### 4.4 叙事原则

- 主角身份不能在开场直接揭示；
- 线索必须允许玩家形成至少两种阶段性解释；
- 结局反转不能否定玩家此前全部经历；
- 关键真相应当能够通过环境证据提前推断；
- 怪物设计、谜题和空间变化都必须服务于剧情主题；
- 不依赖大段旁白补救叙事漏洞。

## 5. 核心玩法循环

### 5.1 基础循环

```text
进入新区域
→ 观察环境并建立空间认知
→ 收集线索或关键道具
→ 发现异常和谜题条件
→ 返回既有区域验证信息
→ 空间发生变化
→ 怪物或环境威胁介入
→ 躲避、逃离或操纵环境
→ 解锁新区域或新时间层
→ 获得剧情碎片
→ 进入下一轮循环
```

### 5.2 典型十分钟体验

1. 玩家从暂时安全的护士站出发；
2. 根据值班表寻找 408 号病房；
3. 发现走廊中没有 408 号，只有 407 和 409；
4. 关闭手电筒后，墙面出现被覆盖的“408”；
5. 玩家进入房间，取得患者录音；
6. 录音播放时，门外响起输液瓶碰撞声；
7. 玩家必须选择躲藏，或关闭录音并从卫生间逃走；
8. 威胁离开后，录音暴露新的档案室密码；
9. 返回护士站时，值班表上出现了主角的姓名。

### 5.3 失败与恢复

- 怪物抓住玩家后从最近检查点恢复；
- 检查点应位于关键谜题完成后；
- 不要求重复长篇阅读和已完成的简单操作；
- 部分失败可以触发短暂记忆片段，提供额外叙事信息；
- 连续失败时可适度降低怪物搜索时间，但不明显提示动态难度。

## 6. 玩家能力与限制

### 6.1 基础操作

| 操作 | 默认按键 |
|---|---|
| 移动 | WASD |
| 视角 | 鼠标 |
| 交互 | E |
| 手电筒 | F |
| 奔跑 | Shift |
| 蹲伏 | Ctrl 或 C |
| 查看物品/线索 | Tab |
| 暂停 | Esc |

所有按键应允许后期重绑定。

### 6.2 移动体验

- 第一人称移动应有重量感，但不能迟钝；
- 不加入复杂跳跃和平台动作；
- 奔跑受短期体力限制；
- 蹲伏用于降低噪声和进入躲藏区域；
- 镜头晃动强度必须可在设置中调整；
- 不使用强制动态模糊作为恐怖手段。

### 6.3 手电筒

手电筒是核心玩法工具：

- 标准照明模式；
- 短暂闪烁和电力干扰；
- 某些痕迹仅在特殊光照条件下出现；
- 光线可能吸引怪物；
- 关闭手电筒可能显现另一类异常；
- 电量系统应制造压力，但避免让玩家永久陷入完全黑暗。

推荐采用“短期过热/不稳定”而不是永久消耗电池的方式，减少无解状态。

### 6.4 物品与证据

物品分为三类：

1. **关键道具：** 钥匙、门禁卡、保险丝、录音带；
2. **证据线索：** 病历、照片、调查文件；
3. **应急道具：** 声源装置、一次性闪光装置、镇静剂。

背包应保持轻量，不设计重量系统和复杂合成。

## 7. 威胁与怪物愿景

### 7.1 主要怪物：值夜护士

#### 视觉概念

- 身形近似人类，但四肢比例存在轻微错误；
- 面部总被过曝白光、纱布或阴影遮挡；
- 穿着不属于同一年代的护士制服；
- 拖着输液架，输液瓶发出可辨认的碰撞声；
- 静止时像场景道具，移动时动作不连续；
- 后期细节逐渐显现其与主角的相似性。

#### 行为原则

怪物依靠声音和视线感知玩家：

```text
巡逻
→ 听到异常
→ 前往调查
→ 发现玩家
→ 追逐
→ 搜索最后已知位置
→ 失去目标
→ 返回或改变巡逻路线
```

#### 公平性原则

- 怪物不能无提示瞬移到玩家身后；
- 不穿墙；
- 不永久知道玩家位置；
- 追逐前应有声音或视觉预警；
- 玩家可以通过学习规则提高生存率；
- 追逐路线必须经过灰盒测试；
- 躲藏不是绝对安全，错误行为会暴露玩家；
- 高难度可以提高感知能力，但不能破坏规则一致性。

### 7.2 非实体威胁

除主要怪物外，还包括：

- 移动的病床和轮椅；
- 会模仿玩家动作的镜像；
- 错误广播引发的空间变化；
- 关闭后重新出现的门；
- 只能在监控画面中看到的人影；
- 会逐渐靠近的病房假人；
- 伪装成安全提示的错误指令。

这些异象主要负责心理压力，不都需要造成死亡。

## 8. 关卡愿景

### 8.1 总体结构

医院由三个主要区域组成，采用“有限枢纽 + 分支探索 + 循环变化”的结构。

#### 一层：门诊与急诊

作用：

- 教学和气氛建立；
- 让玩家学习基础交互；
- 建立医院曾经正常运行的证据；
- 主要使用环境异常，暂不进行完整追逐。

关键区域：

- 门诊大厅；
- 挂号处；
- 急诊室；
- 保安室；
- 药房；
- 消防通道。

#### 二层：住院部

作用：

- 游戏核心区域；
- 循环走廊；
- 怪物首次正式出现；
- 多数关键谜题与人物线索集中于此。

关键区域：

- 401～409 号病房；
- 404 号病房；
- 护士站；
- 治疗室；
- 公共卫生间；
- 储物间；
- 医生值班室。

#### 地下一层：手术区与太平间

作用：

- 事故真相揭示；
- 高压探索；
- 最终谜题与追逐；
- 结局选择。

关键区域：

- 手术室；
- 消毒间；
- 档案室；
- 设备间；
- 太平间；
- 地下通道；
- 旧电梯井。

### 8.2 空间设计原则

- 玩家应能通过独特地标辨认位置；
- 不用完全相同的走廊无限复制；
- 变化前先让玩家熟悉原始布局；
- 谜题路线与逃生路线应有交叉；
- 安全区域不能永远安全；
- 但必须存在短暂缓冲空间；
- 每个主要区域至少有一个可形成记忆的视觉主题；
- 追逐路线应提供至少两个决策点，而不是单线奔跑。

## 9. 谜题愿景

### 9.1 谜题类型

- 环境观察谜题；
- 病历编号与房间对应；
- 声音顺序和广播频率；
- 监控画面与现实空间差异；
- 不同循环之间保留信息；
- 光照条件下显现隐藏内容；
- 物品时间线重建；
- 道德选择与证据取舍。

### 9.2 谜题原则

- 谜题答案必须能在游戏内推导；
- 不依赖玩家搜索外部资料；
- 不大量使用无上下文数字密码；
- 解谜过程应推动剧情或改变空间；
- 失败可以产生代价，但不应永久锁死流程；
- 重要谜题应有分层提示；
- 提示优先通过环境、主角反应或重复线索呈现，而不是直接显示答案。

## 10. 恐怖节奏与事件分级

### 10.1 一级：不安

- 灯管闪烁；
- 远处轮椅移动；
- 门把手转动；
- 不属于玩家的脚步；
- 广播念错姓名。

### 10.2 二级：异象

- 镜中多出人物；
- 走廊尽头出现静止人影；
- 房间物品在转身后改变；
- 电梯打开后仍是走廊；
- 病历内容出现玩家刚做过的事。

### 10.3 三级：直接威胁

- 怪物经过门外；
- 躲藏点被短暂检查；
- 玩家必须关灯或停止移动；
- 安全出口被封闭；
- 怪物根据玩家噪声改变路线。

### 10.4 四级：追逐

追逐数量应有限，并承担不同功能：

1. 第一次：教授逃生和躲藏规则；
2. 第二次：要求玩家利用熟悉地图和环境机关；
3. 最终追逐：结合剧情选择、空间崩坏和最终谜题。

追逐后应设置情绪释放期，避免玩家持续疲劳。

## 11. 美术愿景

### 11.1 推荐风格

采用：

```text
低多边形写实基础
+ PS2 / 早期 PC 恐怖游戏质感
+ 现代动态灯光和声音空间化
```

选择这一方向的原因：

- 与 Python/Panda3D 技术规模匹配；
- 降低模型和材质制作成本；
- 低精度视觉本身能够制造不确定感；
- 更容易形成统一艺术风格；
- 允许将资源集中在灯光、构图和声音上；
- 对中低配置电脑更友好。

### 11.2 色彩语言

| 状态 | 主要色彩 |
|---|---|
| 暂时安全 | 冷白、暗蓝、浅绿 |
| 潜在危险 | 锈黄、低饱和橙、灰黑 |
| 直接危险 | 暗红、深黑、过曝白 |
| 记忆片段 | 低饱和暖色、褪色棕 |
| 真相空间 | 冷白与红色同时出现 |

### 11.3 图像生成工具定位

`gpt-image-2` 可用于：

- 概念艺术；
- 场景气氛参考；
- 海报和封面；
- 墙面海报、旧照片、患者绘画等叙事素材；
- UI 视觉方向探索；
- 低风险装饰纹理草案。

不直接依赖生成式图片制作：

- 必须精确拼接的主场景材质；
- 含关键谜题数字和文字的最终素材；
- 碰撞模型；
- 需要动画骨骼一致性的角色模型。

所有 AI 素材在正式发布前必须检查许可、文字准确性、风格一致性和潜在瑕疵。

## 12. 声音愿景

### 12.1 声音目标

声音应在玩家看到危险前提供信息，并能够独立构成恐怖体验。游戏在戴耳机时应具有最佳体验，但必须提供扬声器可用的混音模式。

### 12.2 声音层级

1. **环境底噪：** 通风、雨声、电流、建筑结构声；
2. **空间音源：** 门外脚步、楼上传来的拖行声、通风管敲击；
3. **玩家反馈：** 脚步、呼吸、衣物摩擦、手电筒开关；
4. **怪物标识：** 输液瓶碰撞、异常鞋跟声、失真呼吸；
5. **叙事音频：** 广播、电话、录音带、监控回放；
6. **音乐：** 少量使用，主要用于章节转折和最终段落。

### 12.3 音频原则

- 避免用突然大音量替代惊吓设计；
- 玩家应能够从左右、远近和遮挡判断音源；
- 怪物接近时可减少环境声音，而不是总增加音乐；
- 相同音效避免机械重复；
- 提供主音量、环境、音效、语音、音乐独立设置；
- 重要语音配字幕，并支持字幕大小调整。

### 13.1 当前技术基础

- Ursina 7.0.0；
- pygame 2.6.1；
- pyglet 2.1.15；
- NumPy、Pillow、matplotlib；
- PyCharm；
- Visual Studio 2022；
- Git 与 GitHub；
- `gpt-image-2` MCP 图像生成能力。

游戏专用 Python 环境：

```text
D:\Users\21036\PycharmProjects\PythonProject\game_env\Scripts\python.exe
```

### 13.2 正式引擎建议

正式项目优先采用 **Panda3D**：

- 提供完整的场景图管理；
- 支持 3D 模型、灯光、碰撞、音频和输入；
- 比基于 Ursina 快速脚本更适合长期架构；
- Ursina 可用于独立机制试验，但不作为正式项目的必要依赖；
- 后续可用 Panda3D 工具链打包 Windows 版本。

### 13.3 技术目标

- 默认目标分辨率：1920×1080；
- 支持窗口、无边框窗口和全屏；
- 目标帧率：中等配置下稳定 60 FPS；
- 最低可接受帧率：30 FPS；
- 支持图形质量预设；
- 支持键位配置和鼠标灵敏度；
- 支持自动存档；
- 游戏逻辑与资源路径分离；
- 关键系统具有日志和调试显示；
- 不将 API Key、GitHub Token 或开发凭据打包进游戏。

### 13.4 技术风险

| 风险 | 应对策略 |
|---|---|
| Panda3D 中文资料少 | 以官方文档、示例和小型技术实验为基础 |
| 3D 资源制作量大 | 采用低多边形风格、模块化场景套件 |
| 怪物 AI 难调 | 先完成可视化状态机和灰盒地图测试 |
| 光影性能压力 | 限制动态灯数量，烘焙静态环境信息 |
| 音频空间感不足 | 尽早测试 3D 音源和遮挡近似方案 |
| 项目规模膨胀 | 固定首发时长和地图范围，新增内容进入候选清单 |
| Python 打包问题 | 在早期技术阶段验证 Windows 可执行文件打包 |

- 动态模糊开关；
- 奔跑方式切换（按住/切换）；
- 字幕开关、大小和背景；
- 关键音效的视觉提示选项；
- 亮度和黑位校准；
- 跳脸强度提示或简化模式（待评估）；
- 追逐难度选项；
- 色觉友好提示，避免只用颜色传达谜题信息。

## 16. 成功标准

### 16.1 体验成功标准

- 玩家在不依赖连续跳脸的情况下感到持续紧张；
- 玩家能够辨认并学习至少三条环境或怪物规则；
- 玩家通关后能够描述主角身份和事故的大致真相；
- 至少一个空间变化能让玩家主动怀疑自己的记忆；
- 声音能实际影响玩家决策；
- 追逐紧张但不被普遍认为不公平；
- 三个结局与玩家收集和选择相关，而非简单菜单选择。

### 16.2 技术成功标准

- Windows 目标设备可以稳定启动和退出；
- 游戏可完成 60～90 分钟流程；
- 自动存档不破坏进度；
- 场景切换无严重卡顿；
- 主要流程不存在阻断通关的 Bug；
- 关键系统能够通过配置文件调整；
- 能生成可发布的独立构建版本；
- 仓库不包含任何 API Key 和个人令牌。

## 20. 愿景总结

《回声病房》是一款规模克制、重视声音和空间记忆的第一人称心理恐怖游戏。它不希望通过无限堆叠怪物和跳脸迫使玩家害怕，而是让玩家在逐渐熟悉医院后，发现自己熟悉的一切都不再可靠。

游戏的最终目标不是让玩家问“怪物在哪里”，而是让玩家问：

> “这一切究竟发生过，还是我为了逃避某件事而重新编造的？”

只要后续设计和制作始终服务于这一问题，本项目就能维持统一、可辨识且可实现的创作方向。
