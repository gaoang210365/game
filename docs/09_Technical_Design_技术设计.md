- 数值/工具：NumPy、Pillow、matplotlib；

- 版本控制：Git + GitHub；
- 图像生成：gpt-image-2 MCP。

### 2.2 游戏专用虚拟环境

```text
D:\Users\21036\PycharmProjects\PythonProject\game_env\Scripts\python.exe
```

### 2.3 引擎选择结论

正式项目采用 **Panda3D**：

- 提供完整场景图、任务、碰撞、音频、输入管理；
- 比基于 Ursina 的快速脚本更适合长期架构；
- Ursina 仅用于独立机制试验，不作为正式依赖；
- 可用 Panda3D 工具链打包 Windows 版本。

不使用 Java。

## 4. 核心模块设计

### 4.1 事件总线（core/events.py）

- 提供发布/订阅机制；
- 各系统通过事件通信，降低耦合；
- 事件示例：玩家产生噪声、门打开、进入区域、触发异象、怪物状态改变、进入新循环层。

### 4.2 游戏状态（core/state.py）

- 管理全局状态（菜单、游玩、暂停、结局）；
- 管理循环层与章节进度；
- 与存档系统交互。

### 4.3 玩家模块（player/）

- controller：移动状态机、噪声输出、视线信息；
- flashlight：照明模式、过热/不稳定、光照信息；
- stamina：奔跑消耗与恢复；
- inventory：关键道具、证据、应急道具。

### 4.4 世界模块（world/）

- level：加载灰盒或正式关卡；
- door：门与容器状态、噪声、撞击；
- interactable：统一交互接口；
- loop：循环触发、空间变化规则、状态记录。

### 4.5 AI 模块（ai/）

- perception：整合声音、光照、视线感知；
- state_machine：巡逻/察觉/调查/追逐/搜索/失去/返回；
- monster：怪物实体，读取参数配置。

### 4.6 系统模块（systems/）

- audio：3D 音源、动态混音；
- horror_events：事件分级调度、节奏控制；
- save：存档序列化与恢复；
- subtitles：字幕显示。

### 4.7 UI 模块（ui/）

- hud：极简提示；
- archive：证据档案；
- menu：主菜单、暂停、设置。

## 5. 数据与配置

### 5.1 配置文件（config/）

- graphics.json：分辨率、窗口模式、质量预设、FOV、镜头晃动、动态模糊；
- controls.json：按键绑定、鼠标灵敏度、奔跑模式；
- audio.json：分类音量、字幕、混音模式；
- game.json：难度、可访问性选项。

### 5.2 关卡数据（assets/levels/）

- 关卡以数据文件描述房间、门、交互点、事件触发器、道具位置、巡逻路径；
- 灰盒阶段可用简单格式（JSON），便于快速调整。

### 5.3 怪物参数

- 集中在配置文件，按难度分组（低/标准/高）；
- 便于灰盒测试调参。

## 6. 存档格式

### 6.1 存档内容

- 玩家位置与朝向；
- 物品与证据收集状态；
- 当前章节与循环层；
- 空间变化状态；
- 事件进度与已触发标记；
- 检查点标识。

### 6.2 规则

- 自动存档 + 章节检查点；
- 采用 JSON 或等价可读格式；
- 存档不包含任何 API Key、Token 或凭据；
- 存档版本号，便于兼容处理。

---

## 7. 性能目标

- 目标分辨率：1920×1080；
- 目标帧率：中等配置稳定 60 FPS；
- 最低可接受：30 FPS；
- 限制动态光源数量；
- 静态环境烘焙信息；
- 资源异步加载，避免主线程阻塞；
- 场景切换无严重卡顿。

## 10. 技术风险与应对

| 风险 | 应对 |
|---|---|
| Panda3D 中文资料少 | 依赖官方文档、示例与小型技术实验 |
| 3D 资源制作量大 | 低多边形、模块化套件 |
| 怪物 AI 难调 | 可视化状态机 + 灰盒测试 |
| 光影性能压力 | 限制动态灯、烘焙静态信息 |
| 音频空间感不足 | 尽早测试 3D 音源与遮挡近似 |
| 项目规模膨胀 | 固定首发范围，新增进候选清单 |
| Python 打包问题 | 早期验证 Windows 可执行打包 |

---

## 11. 打包与发布

- 使用 Panda3D 打包工具或 PyInstaller 等方案验证 Windows 可执行文件；
- 早期技术阶段即测试打包，避免后期风险；
- 发布产物通过 GitHub Release 或外部存储分发；
- 大型资源评估 Git LFS；
- 构建前检查不包含敏感凭据。

---

## 12. 安全与仓库规范

- 独立仓库（建议 echo-ward），不与图床混用；
- .gitignore 排除：saves/、构建产物、临时文件、任何含密钥的本地配置；
- 不提交 API Key、GitHub Token、中转站配置；
- 敏感配置通过本地环境变量或未纳入版本控制的本地文件管理。
